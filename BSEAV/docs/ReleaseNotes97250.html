Build Instructions to create an initrd 7250b0 Linux Kernel Image

  • Extract toolchain, root filesystem and Linux kernel tar balls: 
    • tar -C /opt/toolchains -xf stbgcc-6.3-1.1.tar.bz2
export PATH=/opt/toolchains/stbgcc-6.3-1.1/bin:$PATH
tar -xf rootfs-4.1-1.14.tar.bz2
tar -xf stblinux-4.1-1.14.tar.bz2
cd rootfs
make vmlinuz-initrd-arm

Environment Setup Instructions for Reference Software

  • Assume 7250B0 Linux kernel is already configured and built (i.e. use vmlinuz-initrd-arm). Thus tool chain and rootfs tree already exist.
  • Set LINUX environment variable: 
    • cd rootfs/linux
export LINUX=`pwd`
  • Set up PATH environment variable to point to tool chain: 
    • export PATH=/opt/toolchains/stbgcc-6.3-1.1/bin:$PATH
  • Set up some reference software environment variables for the 7250B0. For additional board specific exports refer to the Section “7250 Supported Boards.” 
    • export B_REFSW_ARCH=arm-linux
export NEXUS_PLATFORM=97250
export BCHP_VER=B0

Build and Usage Instructions for Atlas

  • Extract reference software tar ball: 
    • tar -xf refsw_release_unified_20171117.src.tgz
  • Build instructions 
    • cd BSEAV/app/atlas/build
make install
  • Connect device to TV via HDMI cable
  • Mount the USB/SATA containing the Movie streams as /data/videos
  • Run the following commands to run atlas after extracting the Atlas output image on the target device: 
    • start install
start atlas

Application Notes

(Top)


Prerequisite for the following application notes are:

  • The 7250 Set-Top has been imaged with Bolt and boots to a Bolt command line prompt (i.e. BOLT>).
  • Binary images are hosted on a TFTP server at <TFTP server IP address>
  • The 7250 Set-Top is connected to the same network as the TFTP server over Ethernet

On Bolt First Boot

Please select the appropriate board when you first boot into BOLT if the board does not provide a board ID for auto-board detection. The BCM972502DGL dongle board is an example of a design that does not provide a board ID. Choose the appropriate board when prompted or use the “setboard [boardnum]” BOLT command to set the board type.

Optionally, use the “rts –set [N]” command to program and save the box mode selection. The “rts” command by itself will return a list of supported Box Modes and the currently selected one.

Boot Initrd Kernel from BOLT

  • Obtain an IP address using DHCP:
    • ifconfig eth0 -auto
  • Load and boot the initrd kernal image from a TFTP server:
    • boot <TFTP_server_IP_address>:vmlinuz-initrd-arm

Set Up Non-Initrd Kernel and Root File System on Raw NAND Flash

  • Follow the instructions under the section “Configuring the Kernel and Root File System” in BroadcomReferencePlatformSetup.pdf

Set Up Non-Initrd Kernel and Root File System on eMMC Flash

  • In BOLT, use “show devices” command to confirm that eMMC flash is recognized
  • Boot initrd kernel (see “Initrd Kernel Boot Instructions from BOLT Command Line”)
  • Partition eMMC flash (assuming the device is /dev/mmcblk0): 
    • sgdisk -o /dev/mmcblk0
sgdisk -a 1 -n 1:34:35 -c 1:"macadr" /dev/mmcblk0
sgdisk -a 1 -n 2:36:163 -c 2:"nvram" /dev/mmcblk0
sgdisk -a 1 -n 3:164:16547 -c 3:"kernel" /dev/mmcblk0
sgdisk -a 1 -n 4:16548:17571 -c 4:"splash" /dev/mmcblk0
sgdisk -a 1 -n 5:17572:17699 -c 5:"devtree" /dev/mmcblk0
sgdisk -n 6:17700:`sgdisk -E /dev/mmcblk0` -c 6:"rootfs" /dev/mmcblk0
  • Copy kernel and rootfs into eMMC flash: 
    • mkfs.ext4 /dev/mmcblk0p6
mount /dev/mmcblk0p6 /mnt/hd
cd /tmp
tftp <TFTP_server_IP_address> -g -r vmlinuz-arm
dd if=vmlinuz-arm of=/dev/mmcblk0p3
tftp <TFTP_server_IP_address> -g -r nfsroot-arm.tar.bz2
cd /mnt/hd
tar -xjf nfsroot-arm.tar.bz2
mv romfs/* .
rmdir romfs
  • Reboot and reprogram MAC from BOLT: 
    • macprog 97250 <board_serial_number> <board_rev_number>
  • BOLT kernel boot command 
    • boot flash0.kernel: 'root=/dev/mmcblk0p6 rootwait rw debug mmc_core.removable=0'

Tips on Working with MHL Dongles

  • Use V10 or later version of the BCM972502DGL design, which allows seamless power up over either USB or MHL. Powering over USB enables HDMI output mode. Powering over MHL enables MHL output mode. The design does not support an asymmetric mix of display output and powering method (i.e. USB power and MHL output mode, and vice versa are not supported).
  • When powered over MHL, the micro host USB port is enabled but does not provide power. To hook up USB devices, such as a flash stick or an Ethernet dongle, an externally powered USB hub must be placed in between the BCM972502DGL dongle and the USB device. You may need to use a USB gender changer A-A (i.e. female to female) adapter.
  • To initially seed the flash with a kernel and root file system, follow the instructions described in “Set Up Non-Initrd Kernel and Root File System on eMMC Flash.” Powering up the dongle over USB may be preferable over MHL to avoid having to rely on additional components such as an USB hub, USB adapters, etc. A single USB Y-Cable (e.g. dual USB A to micro B) should be sufficient.
  • EXT4 file systems are typically used for eMMC data partitions. They utilize buffering to improve IO efficiency with relatively long commit cycles. Thus data loss is possible if sudden power loss occurs, such as when a MHL powered dongle is disconnected from a TV. To minimize the potential for data loss, applications should call fflush() at critical points in their flow to ensure data is committed to flash.
  • In a production environment, the MHL interface once initialized by an application can only be reinitialized by transitioning the SoC into and out of S3 and/or a power cycle. For example, if a Nexus application that initialized MHL quits while in S0, MHL will not be usable again until after a S3 transition or power cycle (i.e. unplugging and re-plugging into a TV). This is the expected behavior.
  • The latest MHL embedded firmware should be used. It is bundled as part of the reference software source release under magnum/portinginterface/hdm/src/mhl/mpm_fw.bin. You may use BroadBandStudio4 (BBS) to download the image into the external MPM flash. Contact your FAE for the BBS scripts to program the latest firmware. In future releases, a software driver will be provided to update the firmware over SPI.
  • BCM972502DGL dongle requires a MHL 2 or MHL 3 TV

Setup and Wiring Instructions for the BCM972502DGL

  • These instructions are applicable for connecting a BCM972502DGL dongle to a BCM9SERIAL_ADPT debug adaptor board.
  • Plug the accompanying custom cable into the dongle’s J330, which is a 2x5 pin header. The other end of the cable is split into MPM_UART, CPU_UART, BBS and GND.
  • On the debug adaptor board, plug the MPM_UART header into J32 (UART0), CPU_UART header into J33 (UART1), BBS header into J30 (I2C) and GND into J33 pin 4. Ensure to line up pin 1 on all headers.
  • Use a standard USB AB cable to connect the debug adaptor board to the PC. Two virtual COM ports should be recognized. You may need to update to the latest FTDI USB driver on the host machine. The first one corresponds to UART0/MPM and the second one corresponds to UART1/CPU. MPM UART settings are 921K-8-N-1 (no hardware flow control). CPU UART settings are 115K-8-N-1 (no hardware flow control).

7250 Production Box Modes

(Top)

There are multiple box modes available which contain the RTS to support the different variants and features of this chip.
Information on the available box modes has been moved to a new HTML table.

Additonally more details about each box mode may be found under nexus/platform/97250/docs.

Box Mode selection may be made during run-time by setting the environment variable B_REFSW_BOXMODE to the desired setting (e.g. “export B_REFSW_BOXMODE=1” to select Box Mode 1 at run-time).

This release contains support for HEVC 1080p30/60 decode and display. This feature can be exercised using Nexus playback or Atlas application.
