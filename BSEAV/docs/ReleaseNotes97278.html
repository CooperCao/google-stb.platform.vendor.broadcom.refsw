Important Notes and Limitations

In order to prevent confusion, due to the variety of configurations for different chip part numbers, BOLT bootloader is no longer bundled with the Reference Software release, and should be requested separately. For this release, 97278 was tested with BOLT v1.35. Use of TrustZone will require BOLT v1.36, available early 2018.


In URSR 17.4, Audio/DSP supports MS12 v1.3.1:

7278 audio support in this release includes UDC (AC3/DDP) [Dolby MS12 license required], AAC/AAC-HE, MPEG and PCMWAV audio decode from TS or ES content, as well as Dolby MS12 processing associated with MS12 Configs A, B, C, D. Note that AC4 decoding is not supported in this release.

7278 audio support also includes AAC stereo encoder.

No other codecs are supported on 7278 in this release.

 

Security (NEXUS_SECURITY_SUPPORT) support is preliminary in this release.  BCM7278 uses a new “secv2” API.  Build with NEXUS_SECURITY_SUPPORT=n if necessary.

 

 

HDCP 1.4 is not functional in BCM7278B0. Will be fixed in B1.

 

Nexus Power Management (NEXUS_POWER_MANAGEMENT=y) is supported in this release, but standby (S1,S2,S3) is not.

 

Nexus Video Encoder (VIDEO_ENCODER_SUPPORT=y) is defaulted on for use with Box Modes 1 and 3 in this release. Transcoding with audio is supported but has received limited testing. A current transcoding limitation is shown below in "Known Issues".

 

Box Mode 4 is disabled due to non-working transcode.

 

HDR 2.0 features supported in this release:

·         HDR Metadata Extraction

·         HDR10 pass-through

·         HLG pass-through

·         SDR --> HDR conversion

·         HDR10 --> SDR conversion

·         HLG --> HDR10 conversion

 

SAGE (SAGE_SUPPORT) is preliminary in this release. Defaulted off. Build with SAGE_SUPPORT=y to enable. Live TS not working with SAGE.

Quick Build Instructions

  • # tar -xvjf rootfs-4.9-1.2.tar.bz2
  • # tar -xvjf stblinux-4.9-1.2.tar.bz2
  • # cd rootfs
  • # make vmlinuz-initrd-arm (For 32-bit version of the kernel build)
  • # make vmlinuz-initrd-arm64 (For 64-bit version of the kernel build)
  • # cd ../linux
  • export LINUX=`pwd`
  • tar -C /opt/toolchains -xvjf stbgcc-6.3-1.1.tar.bz2
  • PATH=/opt/toolchains/ stbgcc-6.3-1.1/bin:$PATH
  • export B_REFSW_ARCH=arm-linux
  • or   export B_REFSW_ARCH=aarch64-linux (For compiling the code with 64 bit toolchain)

Quick Boot instructions

  • BOLT> boot -z <network_location>:49-1.2/vmlinuz-initrd-arm
  • BOLT> boot -z <network_location>:49-1.2/vmlinuz-initrd-arm64  - This command will boot 64 bit kernel instead of 32 bit kernel.

 

Set the kernel to autoboot

 

  • BOLT> flash <network_location>:49-1.2/vmlinuz-initrd-arm flash0.kernel

·         Or BOLT> flash <network_location>:49-1.2/vmlinuz-initrd-arm64 flash0.kernel  - For 64 bit kernel

  • BOLT> setenv -p STARTUP "boot -z flash0.kernel:"
  • BOLT> reboot

 

How to set a board to autoboot with kernel and ubifs rootfs and binaries into the flash:

·         Boot with vmlinuz-initrd kernel.

·         Run "stbutil"

·         Select option 8 for flashing the noninitrd kernel and rootfs to the eMMC flash.

·         Reboot the board with "boot emmcflash0.kernel: 'root=/dev/mmcblk0p6 rootwait rw bmem=582m@2490m bmem=572m@13764m brcm_cma=256m@13508m'"

·         Set the STARTUP or a function key variable to "boot emmcflash0.kernel: 'root=/dev/mmcblk0p6 rootwait rw bmem=582m@2490m brcm_cma=256m@2232m bmem=572m@13764m brcm_cma=256m@13508m'"

·         Reboot the board.


Note:  The bmem and brcm_cma are required for the brcmv3d driver, to allow the MMU for the 3D graphics core to access memory.  If 3D graphics is not used,
those statements can be omitted from the boot command.  They can be adjusted as recommended by messages during Nexus boot.  The example given is adjusted to work for both Box Mode 1 and Box Mode 2.  If 3D graphics is used with Box Mode 2, the brcm_cma=256m@2232m is useful. For other box modes, it may be removed. See BSEAV/lib/gpu/docs/VC5_AppNote_MMU.pdf for more information.

For BCM97278IPA, with "ZD" chip for internal development, a board issue causes the "v7-32" global address map to be selected. If you don't see the message "v7-64 Memory Map Variant detected." and encounter heap allocation problems during Nexus initialization, an alternative set of boot parameters, 'bmem=582m@1466m brcm_cma=256m@1208m bmem=572m@3524m brcm_cma=256m@3268m' may be needed.

Important: brcm_cma regions must be 4MB aligned.

Instructions to build nexus playback utility and use it to decode and display a UHD stream

  • # tar xvfz refsw_release_97278_20180104.src.tgz
  • #cd nexus/utils
  • #make install
  • Connect UHD 4K TV to HDMI
  • #nexus playback -display_format 3840x2160p60 -component off -max_video 3840,2160 -probe <stream_name >

 

 

Instructions to build atlas and use it to play UHD stream

  • # tar xvfz refsw_release_97278_20180104.src.tgz
  • #cd BSEAV/app/atlas/build
  • #make install
  • Connect UHD 4K TV to HDMI
  • Mount the USB/SATA containing the 4kX2k stream as /data/videos
  • Run the following commands to run atlas after extracting the tarball from obj.97278/BSEAV/bin dir.
  • #start install
  • #start atlas
  • From the menu, select Playback -> 4kp60 stream.
  • Using the remote control for the board, select Menu -> Display -> Format -> 2160p 60Hz. You should be able to see the stream playback in 4Kp60 display format.
