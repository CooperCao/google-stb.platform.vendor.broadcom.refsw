Important Notes and Limitations

In order to prevent confusion, due to the variety of configurations for different chip part numbers, BOLT bootloader is no longer bundled with the Reference Software release, and should be requested separately. For this release, 97278 was tested with BOLT v1.38.

 

In URSR 18.1, Audio/DSP supports MS12 v1.3.1:

7278 audio support in this release includes UDC (AC3/DDP), AC4 [Dolby MS12 license required], AAC/AAC-HE, MPEG and PCMWAV audio decode from TS or ES content, as well as Dolby MS12 processing associated with MS12 Configs A, B, C, D.

7278 audio support also includes AAC stereo encoder.

No other codecs are supported on 7278 in this release.

 

BCM7278 uses a new “secv2” API.  Build with NEXUS_SECURITY_SUPPORT=n if necessary.

 

HDCP 1.4 is not functional in BCM7278B0. Fixed in B1.

 

Nexus Video Encoder (VIDEO_ENCODER_SUPPORT=y) is defaulted on for use with Box Modes 1 and 3 in this release.

 

HDR 2.0 features supported in this release:

·         HDR Metadata Extraction

·         HDR10 pass-through

·         HLG pass-through

·         SDR --> HDR conversion

·         HDR10 --> SDR conversion

·         HLG --> HDR10 conversion

Release Notes for ASP in URSR 18.1

 

Steps to enable ASP support in NEXUS:

 

Steps to build:

1.     Set following addition build flag:

export NEXUS_ASP_SUPPORT

 

2.     Run plat script as following:

source BSEAV/tools/build/plat 97278 B1

and make sure following environment variables get set:

NEXUS_PLATFORM=97278

BCHP_VER=B1

B_REFSW_DEBUG=

B_REFSW_ARCH=aarch64-linux

B_REFSW_OS=linuxuser

LINUX=/opt/brcm/linux-4.9-1.4/arm64

Toolchain found at /opt/toolchains/stbgcc-6.3-1.2/bin/aarch64-linux-gcc

 

3.     Build the ASP Driver, it will be copied in the obj.97278.B1/nexus/bin directory!

cd BSEAV/linux/driver/asp

make all

 

4.     Build http_file_streamer example app, it will build nexus & all needed libraries into obj.97278.B0/nexus/bin directory

cd nexus/lib/bip/examples/http_file_streamer

make all

 

Steps to run:

1.     Ensure BOLT version of 1.37 is flashed on the 97278 board.

 

2.     Use the 4.9-1.4 Linux kernel. The CONFIG_NETFILTER option must be enabled.

 

3.     Mount the nexus/bin directory via NFS or copy all the binary files to the STB.

 

4.     Copy any media files into the /data/videos directory of the STB.

 

5.     Connect the STB to a local area network via the gphy Gigabit Ethernet port and have one or more network clients (STB, Linux or Windows PC) also connected on that network.

 

6.     Run http_file_streamer app w/ following options (use –h option to see the help):

nexus http_file_streamer -enableHwOffload –loop -inactivityTimeoutInMs 12000 -dontAddContentLength -enableAllPass -maxDataRate 40000000 -disableTrickmode

 

7.     From a network client (STB, Linux or Windows PC), run either wget or an ip player (nexus/lib/bip/example/player example) to pull the content from the STB streamer:

wget –O <outputFile.ts> http://<Server’s IP>:<Server’s Port>/<Target file name>

e.g. wget –O test.ts http://192.168.2.10:81/test.ts

 

Supported Features:

·         Streaming out of content from Hard Drive, Tuner, or IP frontends to network clients w/o DTCP/IP encryption.

·         Supported media formats: MPEG2 TS

·         Supported Max throughput via ASP: 640Mpbs

·         Max ASP Channels: 16

·         Nexus build mode: User Mode, NxClient Mode

 

Pending Features (still under development):

·         Streaming of DTCP/IP Encrypted content via ASP

·         Stream-in: receiving of incoming stream via ASP

·         Streaming out of non-MPEG2 TS containers including HLS, MPEG DASH, etc.

·         Playback from local host using ASP

·         Nexus in Kernel mode

 

Quick Build Instructions (general)

  • # tar -xvjf rootfs-4.9-1.4.tar.bz2
  • # tar -xvjf stblinux-4.9-1.4.tar.bz2
  • # cd rootfs
  • # make vmlinuz-initrd-arm (For 32-bit version of the kernel build)
  • # make vmlinuz-initrd-arm64 (For 64-bit version of the kernel build)
  • # cd ../linux
  • export LINUX=`pwd`
  • tar -C /opt/toolchains -xvjf stbgcc-6.3-1.2.tar.bz2
  • PATH=/opt/toolchains/ stbgcc-6.3-1.2/bin:$PATH
  • export B_REFSW_ARCH=arm-linux
  • or   export B_REFSW_ARCH=aarch64-linux (For compiling the code with 64 bit toolchain)

Quick Boot instructions

  • BOLT> boot <network_location>:49-1.4/vmlinuz-initrd-arm
  • BOLT> boot <network_location>:49-1.4/vmlinuz-initrd-arm64  - This command will boot 64 bit kernel instead of 32 bit kernel.

 

How to set a board to autoboot with kernel and ubifs rootfs and binaries into the flash:

·         Boot with vmlinuz-initrd kernel.

·         Run "stbutil"

·         Select option 8 for flashing the noninitrd kernel and rootfs to the eMMC flash.

·         Reboot the board with "boot emmcflash0.kernel: 'root=/dev/mmcblk1p6 rootwait rw bmem=640m@1404m bmem=824m@12488m brcm_cma=128m@1276m brcm_cma=128m@12360m'"

·         Set the STARTUP or a function key variable to "boot emmcflash0.kernel: 'root=/dev/mmcblk1p6 rootwait rw bmem=640m@1404m bmem=824m@12488m brcm_cma=128m@1276m brcm_cma=128m@12360m'"

·         Reboot the board.

 

Note:  The bmem and brcm_cma are required for the brcmv3d driver, to allow the MMU for the 3D graphics core to access memory.  If 3D graphics is not used,
those statements can be omitted from the boot command.  They can be adjusted as recommended by messages during Nexus boot.  The example given is adjusted to work for both Box Mode 1 and Box Mode 2.  If 3D graphics is used with Box Mode 2, the brcm_cma=256m@2172m is useful. For other box modes, it may be removed. See BSEAV/lib/gpu/docs/VC5_AppNote_MMU.pdf for more information.

For BCM97278IPA, with "ZD" chip for internal development, a board issue causes the "v7-32" global address map to be selected. If you don't see the message "v7-64 Memory Map Variant detected." and encounter heap allocation problems during Nexus initialization, an alternative set of boot parameters, 'bmem=640m@1404m bmem=824m@3272m brcm_cma=128m@1276m brcm_cma=128m@3144m' may be needed.

These suggested bmem parameters were chosen as a set of "worst case" values that can be used with all box modes. You may be able to find bmem and brcm_cma values more optimal for your application. Try running with none, then look for the "nexus_platform_cma: recommended Linux boot parameters:" message during Nexus initialization. Note that the example values allow some space at the top of the first bmem region for a reserved region 'NWMBOX'.

Important: brcm_cma regions must be 4MB aligned.

Instructions to build nexus playback utility and use it to decode and display a UHD stream

  • # tar xvfz refsw_release_unified_20180301.tgz
  • #cd nexus/utils
  • #make install
  • Connect UHD 4K TV to HDMI
  • #nexus playback -display_format 3840x2160p60 -component off -max_video 3840,2160 -probe <stream_name >

 

 

Instructions to build atlas and use it to play UHD stream

  • # tar xvfz refsw_release_unified_20180301.tgz
  • #cd BSEAV/app/atlas/build
  • #make install
  • Connect UHD 4K TV to HDMI
  • Mount the USB/SATA containing the 4kX2k stream as /data/videos
  • Run the following commands to run atlas after extracting the tarball from obj.97278/BSEAV/bin dir.
  • #start install
  • #start atlas
  • From the menu, select Playback -> 4kp60 stream.
  • Using the remote control for the board, select Menu -> Display -> Format -> 2160p 60Hz. You should be able to see the stream playback in 4Kp60 display format.

 

 

Using splashgen

·  Because the BOLT bootloader is 32-bit, graphics surfaces used for splash screen must be in MEMC0 (If v7-64 global address map is in use). To insure that, you can run with the run-time environment setting

B_REFSW_BOXMODE=2

·  If that's not possible, modify the BSEAV/app/splash/splashgen/97278/bsplash_board.h to set 'SPLASH_SURF0_MEM' and 'SPLASH_SURF1_MEM' to '0' (numeric zero) so that only the first memory controller is used when generating and running splash.
