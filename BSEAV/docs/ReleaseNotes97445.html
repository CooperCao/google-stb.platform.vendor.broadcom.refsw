BSECK-A/B parts:

·         Recently, the security OTP on 7445 chip has been updated to use BSECK-B FW in the bolt. Now we have two different types of parts in use. BSECK-A and BSECK-B programmed parts. Bolt binaries are different for the two types of parts. You must use matching bolt binaries in order to boot bolt. Separate bolt binaries are provided for each type of parts.

·         The Bolt release v0.93 supports both BSECK-A and BSECK-B parts. SW changes in Kernel/Nexus/PI are not required. A separate Bolt image is needed for BSECK-A /-B parts, otherwise the Bolt/board won’t boot. There are two ways to identify the BSECK-A/B part on a board, 

            1) Read the label of chip p/n next to the silicon on the board or the label on the chassis (rear panel). 

If the label is “BCMxxxxZBKFSBAxx” meaning BSECK-A part, the Bolt image: bolt-<VERSION>-<FAMILY>-ba-bfw-<X.Y.Z>.bin should be used. 

If the label is “BCMxxxxZBKFSBBxx” meaning BSECK-B part, the Bolt image: bolt-<VERSION>-<FAMILY>-bb-bfw-<X.Y.Z>.bin should be used. 

2) Check out the BSP_GLB_CONTROL_OTP_HASH_CRC register (physical address: 0xf032d060) in case the label cannot be found. 

If the register value = 0x7cf57343, it means ZB BSECK-A part and the Bolt image: bolt-<VERSION>-<FAMILY>-ba-bfw-<X.Y.Z>.bin should be used. 

If the register value = 0x06272a51, it means ZB BSECK-B par and the Bolt image: bolt-<VERSION>-<FAMILY>-bb-bfw-<X.Y.Z>.bin should be used 

  

7445 bond outs such as BCM7252ZB/7444ZB/7448ZB/7449ZB with the same BBL/BSECK version has the same Hash CRC value. The instructions here are just for ZB part, please work with Broadcom security support team for the bootloader of CA parts.

  • # tar -xvjf rootfs-4.1-1.10.tar.bz2
    rootfs-4.1-1.10.tar.bz2
  • # tar -xvjf stblinux-4.1-1.10.tar.bz2
  • # cd rootfs
  • # make vmlinuz-initrd-arm
  • # cd ../linux
  • export LINUX=`pwd`
  • tar -C /opt/toolchains -xvjf stbgcc-4.8-1.6.tar.bz2
  • PATH=/opt/toolchains/ stbgcc-4.8-1.6/bin:$PATH
  • export B_REFSW_ARCH=arm-linux

 

Quick Boot instructions

  • BOLT> boot -z <network_location>:4.1-1.10/vmlinuz-initrd-arm

 

Set the kernel to autoboot

 

  • BOLT> flash -noheader <network_location>:4.1-1.10/vmlinuz-initrd-arm flash1.rootfs0
  • BOLT> setenv -p STARTUP "boot -z flash1.rootfs0:"
  • BOLT> reboot

 

How to set a board to autoboot with kernel and ubifs rootfs and binaries into the flash:

·         Boot with vmlinuz-initrd kernel.

·         Run "stbutil"

·         Select option 1 for flashing the noninitrd kernel to the flash. Again run "stbutil" and select 2 to flash rootfs into the flash.

·         Reboot the board with "boot flash0.kernel 'ubi.mtd=flash1.rootfs0 rootfstype=ubifs root=ubi0:rootfs rw'"

·         In case you want to run any application on the startup then, copy the application binaries in /data/bin of the flash. Copy the streams in /data/media. Edit /etc/profile and put the commands you want to run after the boot. Save the file.

·         Set the STARTUP variable to "boot flash0.kernel 'ubi.mtd=flash1.rootfs0 rootfstype=ubifs root=ubi0:rootfs rw'"

·         Reboot the board.

Instructions about SAGE and HDCP 2.2

·         SAGE is a security core that is involved in various security software such as DRMs and HDCP 2.2.     

·         In order to enable SAGE for DRMs that can take advantage of it (playready 2.5, Netflix and Adobe 5.0), add the following environment variable before building nexus:

export SAGE_SUPPORT=y

·         Also this environment variable should remain when building an application that uses such DRM.

·         HDCP 2.2 Transmit support is only available on 97445 D0 and 97252 D0.

·         To enable it on those platforms, add the following in the environment variables before building nexus:

export SAGE_SUPPORT=y

export NEXUS_HDCP_SUPPORT=y

·         In order to use Hdcp 2.2 feature you also need a proper DRM bin file for the HDCP keys. Please contact Broadcom to receive the proper DRM Utility and instructions.

 

Instructions about how to exercise the power management features:

·         Please refer to nexus/docs/RefswPowerManagementOverview.pdf for the information on Power management support.

·         To exercise the power management states (s0, s1, s2& s3), try BSEAV/app/standby/standby.c and nexus/examples/power_down.c apps. (Make sure you have the board Rev#12 or higher to try S3 mode)

·         Dynamic power management feature is enabled in nexus by default and can be turned off by export NEXUS_POWER_MANAGEMENT=n.

·         With the released kernel(4.1-1.10), S1(Active standby) mode testing is not supported yet.

·         With the released kernel(4.1-1.10), use ‘pml’ script to test various power management modes and to set the wakeup device (timer)(The default wakeup timer will be set to 1 second.).

 Run “pml --help” for more information.

·         With the released kernel(4.1-1.10), S2(Passive standby) mode can be exercised as follows:

#pml

·         With the released kernel(4.1-1.10), S3(Deep Sleep standby) warm boot can be exercised as follows: (Make sure you have the board Rev#12 or higher to try this mode)

#pml -d

·         With the released kernel(4.1-1.10), S5(S3 cold boot) mode can be exercised as follows:

#pml -p
