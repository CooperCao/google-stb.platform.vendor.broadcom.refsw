System Requirements

    Operating Systems:

This package includes software required to exercise basic operation of the 4325-based SDIO dongle, with various features. It includes components for the low power embedded device (dongle) as well as components for the host processor to which the dongle device is connected. This distribution supports the Linux 2.4 and Linux 2.6 kernels, and includes precompiled host binaries for x86 systems.

    Hardware:

Hardware Requirements for the SDIO Dongle:

  • PC with RedHat 9 Linux (2.4 kernel) or Fedora Core 4 (2.6 kernel).
  • Arasan PCI SDIO Host Controller, Rev 1.2c or 1.2c2
  • SDIO Card to Mictor adapter board (BCM9SDIO2CONAD) 
  • Blue Mictor ribbon cable 
  • SDIO Dongle board

How To

Installation

Hardware Installation for SDIO:

  • Install the Arasan card in an available PCI slot; make sure its bracket is screwed into the PC chassis.
  • Insert one end of the blue Mictor ribbon cable into the SDIO Card to Mictor Adapter board (BCM9SDIO2CONAD). Either end of the cable will work, but the connectors are keyed, and will only fit one way. Make sure the connector is fully seated.
  • Insert the Adapter in the SD Connector on the Arasan card. Note that only on SD connector is operational: on a Rev 1.2c card use the connector furthest from the PCI pins; on a Rev 1.2c2 card use the connector closes to the PCI pins. The gold fingers of the Adapter board should face the Arasan Controller PC Board. If the board pops out of the SD card slot repeatedly, then the SD connector is faulty and needs to be replaced.
  • Insert the other end of the Mictor cable into the Apollo adapter board. Again, make sure it is fully seated.

Description of Modules

dhd.o, dhd.ko

Dongle Host Driver. This is a linux kernel module that includes software to drive both the Arasan Host Controller card and the BCM4325SDGWB card, presenting an Ethernet interface to the Linux kernel.

dhd

An executable application which can be used to download a dongle software image to the SDIO dongle via the dhd.o module, and provides other access to dhd.o module controls for test and debug.

wl

An executable application used to control the wireless software on the dongle, e.g. initiate associations and modify WLAN parameters.

dongle images

There are two binary images provided for download into the dongle itself:

  • sdio-arm-thumb-sb-run-noretail-g-cdc-reclaim-idsup
  • sdio-arm-thumb-sb-run-noretail-g-cdc-reclaim-idsup-pfn-aoe-toe

The first provides standard Broadcom 802.11g WLAN functionality, including an in-driver supplicant for WPA-PSK, WPA2-PSK; the second adds several offload features specifically for use in the dongle: Preferred Network Offload, Arp Offload, and TCP Checksum Offload.

Operation

Software Operation for SDIO (do all the following as root):

  • insmod dhd.o (2.4 kernel) or dhd.ko (2.6 kernel)
  • dhd download binfile
    where binfile is a path to one of the dongle images
  • ifconfig ethN [ipaddress] up
    where ethN is the interface for dhd.o
  • wl up 
  • wl scan 
  • wl scanresults 
  • wl join 
  • wl assoc 

Before unplugging the BCM94325 board: 

  • ifconfig ethN down 
  • rmmod dhd 

Compilation

Although the dongle images are provided in binary form only, all the host images provided may be rebuilt from sources.

To rebuild the dhd kernel module:

  • % cd src/dhd/linux
  • % make dhd-cdc-sdstd

To rebuild the dhd application:

  • % cd src/dhd/exe
  • % make

To rebuild the wl application:

  • % cd src/wl/linux
  • % make

Software Overview

This is a host code sampling drop. It is meant to provide a vehicle to demonstrate BCM4325 functional capabilities, and illustrates the host software structure. Future source drops for general use are expected to be based on a different development branch intended to improve portability and host software efficiency.

Current Known Limitations, Problems and Notices 

  • This software snapshot has not undergone full testing.
  • The SD bus clock must currently remain always on.
  • Explicit stack entry points for MAC address change and multicast registration are not included.
  • Host driver may be unstable if the SDIO card is removed unexpectedly (without rmmod first).
  • Host driver code is fully synchronous including spinning waits, resulting excessive CPU utilitization.
  • Operating system dependencies in the dhd module are not sufficiently isolated.

File Organization

The software provided in binary form (described above) is found in the following subdirectories:

  • dongle_images/
    Binary images for download to the dongle
  • modules/
    Linux kernel modules (dhd.o or dhd.ko)
  • apps/
    User-level applications dhd and wl

The source code is found in the src/ directory, and is organized as follows:

  • src/bcmsdio/sys/
    Software for SD bus access. Files named bcmsdstd* implement a driver for standards-compliant SD controllers (developed and tested against the Arasan cards described above); files named bcmsdh* provide a "glue" layer, a placeholder to allow implementing the same client interface on top of other host controller software APIs.
  • src/dhd/sys/
    Software implementing the Dongle Host Driver, targeted toward the bcmsdh interface. Drives the overall dongle, presenting an Ethernet interface to the host network stack and transferring frames to and from the dongle.
  • src/dhd/linux/
    Build directory for the dhd driver.
  • src/dhd/exe
    Source and build directory for the dhd application.
  • src/wl/exe
    Source and build directory for the wl application.
  • src/shared/
    Source files not specific to a single driver or application.
  • src/include/
    Most (cross-directory) header files.
  • src/dongle/
    Dongle-specific header file.

Offload Feature Usage

Preferred Network Offload (PNO)

With PNO enabled, the dongle periodically initiates scans for WLAN networks. The host may specify an explicit list of SSIDs, and/or indicate an interest in all SSIDs. The device will generate an event to notify the host when an applicable SSID is found, or when a previously found SSID has not been seen for a specified amount of time.

An AutoConnect option allows the dongle to initiate WLAN associations without host intervention; the dongle will automatically connect to the \223best\224 SSID in the specified list -- meaning either the first specified (listorder), or the one with the strongest signal (rssi order). If AutoSwitch is also enabled, the dongle will initiate a new connection of the \223best\224 available changes (e.g. if an available BSS has a signal strength exceeding that of the current association by a specified amount in rssi order.)

  • wl pfnset [parameter value] ...
    Sets global PNO operating parameters, one or more of:
    • scanfreq secs
      How often to scan (default 30)
    • netimeout secs
      Timeout to declare a silent AP gone (default 60)
    • sort listorder|rssi
      Ranking method for AutoConnect/AutoSwitch (default listorder)
    • autoswitch 0|1
      Disable/Enable AutoSwitch (default 0)
    • rssi_delta
      Minimum dbm improvement to trigger AutoSwitch (default 30)
    • bkgscan 0|1 Report on all SSIDs, not just the list (default 0)
  • wl pfn 0|1
    Globally enable or disable PNO operation.
  • wl pfnadd ssid [attribute value] ... Add SSIDs to the notification (or autoconnect) list, attributes include:
    • key keystring
      Key for autoconnect to a PSK network
    • imode bss|ibss
      Infrastructure or adhoc BSS (default bss)
    • amode open|shared
      802.11 authentication type
    • wpa_auth wpadisabled|wpapsk|wpa2psk|wpanone
      Type of WPA authentication if applicable
    • wsec WEP|TKIP|AES|TKIPAES
      Type of encryption used
  • wl pfnclear
    Clear the list of SSIDs.
  • wl pfneventchk Display selected (PNO-related) events (for test/demo purposes)
  • wl event_filter Request forwarding of PNO-related events (for test/demo purposes)
PNO Notes
  • Must use pfnset before enabling with pfn
  • PNO must be disabled in order to use pfnset, pfnadd, or pfnclear.
  • The AutoConnect feature is fixed on (not configurable) in this package.
  • Most standard scan and join parameters of WL apply to PNO scans and joins.
  • New events for PNO are WLC_E_PFN_NET_FOUND and WLC_E_PFN_NET_LOST, event numbers 33 and 34 respectively.

Arp Offload (ARPO)

ARPO allows the dongle to discard ARP broadcasts not intended for its host. The device may be configured to respond to ARP requests for the host address, and/or reply to host requests for remote IP where possible. (The device snoops ARP replies to learn remote addresses and caches up to 16 addresses in its own ARP table, with a configurable timeout.) Multi-homing hosts are supported with up to 8 local IP addresses; local addresses may be explicitly configured, or the device may be directed to snoop outbound ARP requests to obtain them.

  • wl arpoe [0|1]
    Enable or disable overall ARP-related features.
  • wl arpoe_ol [[+|-]option] ...
    Set arp-related offload options; allows an option to be added (+) or removed (-) from the current set. Options:
    • agent
      Must be enabled to allow any ARP filtering
    • snoop
      Learn host addresses from outbound ARP requests
    • host_auto_reply
      Reply to host if requested IP is known in table
    • peer_auto_reply
      Reply to network requests for a host address
  • wl arp_hostip [ipaddress]
    Explicitly add a local IP address (required if snooping is not used).
  • wl arp_hostip_clear
    Clear all local host IP addresses from the dongle.
  • wl arp_peerage [secs]
    Set the timeout for remote addresses in the dongle ARP table (default 1200).
  • arp_table_clear
    Clears all remote addresses from the device ARP table.
  • arp_stats
    Displays statistics (number of packets filtered, and so forth)
  • arp_stats_clear
    Clears the statistics counters
ARPO Notes
  • Until the first local address is configured or learned, all received packets are forwarded to the host (not filtered).
  • If snooping is enabled, requests for unknown addresses are forwarded to the host not filtered).
  • All link-local probes (ARP requests with source IP 0.0.0.0) are forwarded to the host (not filtered).

Checksum Offload (CHKO)

CHKO allows the dongle to generate and verify IP, TCP, UDP, and ICMP checksums in non-fragmented IP packets. Checksum generation on transmit and checking on receive are independently configurable. Note that a packet received with a bad checksum is forwarded to the host without the checksum-verified indication; the device does not discard bad packets, and does not explicitly notify the host of checksum failure.

  • wl toe [0|1]
    Enable or disable use of TCP/IP-related offload features.
  • wl toe_ol [[+|-]option]
    Set tcp-related offload options; allows an option to be added (+) or removed (-) from the current set. Options:
    • tx_csum_ol
      Enable checksum generation on transmit
    • rx_csum_ol
      Enable checksum verification on receive
