The BCM47XX Resource Guide

Introduction

In a typical BCM47XX customer engagement, more than just silicon is provided to the customer.  Additional resources such as full residential gateway source implementations, simple bridge implementations, supporting build tools, supplemental debug facilities and pre-tested platform binaries may also be provided.  The purpose of the BCM47XX Resource Guide is to get the customer introduced to the packages and tools that are included in a release.

Supported Reference Platforms

Several sections of this document reference BCM4710-based reference platforms.  Two Broadcom-supplied reference platforms are supported by this document:  the BCM94710R4 and the BCM94710D.  The BCM94710R4 is a compact router design with 4 switched 10/100 Mb LAN Ethernet ports, a 10/100 Mb WAN Ethernet port and an HPNA LAN port.  The BCM94710D development boards replaces the 4 switched Ethernet ports with a single 10/100 Mb Ethernet port, has increased memory size, a serial port, a PCMCIA port and 2 PCI slots.

Release Packages

There are a range of possible release packages for to support BCM4710-based designs:  A release may include either the VxWorks or Linux operating system; It may contain source code or only binary modules; It may include only drivers and BSP or it may include full router source code.  Because of the variety of release packages, certain sections of the resource guide may not be applicable to your particular project.

 

The BCM947XX Linux Package

Introduction

The Linux package may be either a simple bridge or a full-fledged router implementation. 

 

The simple bridge package will include the following features:

 

  • Allow packet bridging between the LAN Ethernet interface and the HPNA interface
  • A command shell accessible via the serial UART connector
  • An extensive list of basic UNIX command-line utilities
  • The iptables package to provide basic routing services

 

In addition to the features of the simple bridge, the full router build includes:

  • Support of dynamic and static IP addresses
  • Supports PPPoE connections on the WAN port
  • Built-in firewall to protect local PCs from outside intrusion
    • Stateful Packet Inspection (SPI) and IP masquerading
  • Router is configurable through any networked PC’s web browser
    • Remote upgrade and remote management
  • Can act as either a DHCP server (for LAN PCs) or a DHCP client (on the WAN port)
  • Compatible with all standard Internet applications
  • Administrators can block specific LAN users internet access
    • IP packet filtering
  • TCP/UDP port forwarding and DMZ capability

 

What You Get

For both the bridge and router packages you will be delivered:

 

  • A general readme file that describes the tools used in building the package, how to install the tools, etc.
  • Source to the Linux kernel in the /src/linux/linux directory
  • Source for all of the Broadcom-supplied components in the root directory:
    • The /components/et/ directory contains source for the Ethernet driver
    • The /src/il/ directory contains source for the HPNA 2.0 driver
    • The /src/shared/ directory contains source code shared among many of the Broadcom-supplied components
    • The /src/include/ directory contains header files for many of the Broadcom-supplied components
    • The /src/router directory contains the remaing bridge/router source modules
  • The /bin directory contains pre-compiled binaries for the package.  The image is a gzipped flat binary image with the appropriate TRX header attached.  The TRX header allows you to use the PMON boot loader to upload the image to your BCM47XX-based router.  If you are not using the PMON boot loader, you can build a new router image and use one of the intermediate file formats (ELF, self-booting, flat binary, etc.) for your purposes.
  • The /doc directory contains this document as well as other supporting documentation
  • The /tools direcotory contains things necessary to build the router image.  This includes:
    • The GCC 3.0 compiler
    • glibc library
  • The pmon/ directory contains source and a binary file for the PMON boot loader      

 

In addition, a full router release package will also include source code for the services necessary to implement a full residential gateway application.

 

Please note that depending on the package that was agreed upon, some or all of the aforementioned source directories may be omitted and replaced with corresponding binary files.

Tools Installation

Along with the Linux distribution, you received compilers, libraries, etc. necessary to build the Linux image for the BCM4710.  The tools are based on gcc 3.0 and make use of glibc version 2.2.3.  Please refer to the README.TXT file in the root of the installation package for information on installing and configuring these tools.

The Linux Kernel

The Linux image is based on freely available Linux kernel source code.  The specific version of the Linux kernel used is 2.4.5.

 

Because of memory and Flash constraints, certain kernel services are not compiled when the BCM94710 Linux image is created.  These services were not necessary in order to provide the desired router functionality.  If, however, you are going to be adding extra router functionality you may need to modify the kernel build in order to include some services that may be necessary to you.  In order to do this, you will need to perform the following:

 

  • Create your own kernel .config file.  This is normally done by executing the command ‘make config’ in the src/linux/linux directory
  • Rename the resulting .config file to something like ‘myconfig’
  • Copy the ‘myconfig’ file to the src/linux/linux/arch/mips directory
  • Copy the file src/Makefile to src/Makefile.bak
  • Open the src/Makefile with your favorite editor
  • Change all instances of the string ‘defconfig-bcm94710-r4’ to ‘myconfig’

 

Now, when you perform the build process (see below), it will use your custom config file instead of the default Broadcom config file.

Adding Additional Services

  • By default, the build will include a number of services to provide functionality to the bridge/router.  The starting of these services is controlled by the Linux shell script located in src/router/misc/rcS in the source distribution.

 

If there are additional services that you would like started when the kernel boots, commands to start them should be added to this startup script.

 

In addition, you will need to include the binaries for your services in the filesystem.  The compiling of your servicescan be integrated into the main router build by editing the makefile/build scripts.  Alternatively, you can build your services separately and then have them copied into the filesystem when the rest of the router is built.

Building the Linux Image

In the top-level of the distribution, there is a makefile.  Once all of the tools have been properly installed, you can simply type ‘make’ in the src/ directory and the PMON boot loader as well as the Linux image will be created.  Upon successful completion of the make command, the PMON boot loader binary will reside in the bootrom/ directory and the Linux binary will reside in the bin/ directory.

 

The Linux kernel will be called linux.trx.  It will contain both the Linux kernel and the root file system and already contain the TRX header necessary for use with the PMON TFTP function.

 

The PMON binary will not have the necessary EST header necessary for programming with the visionICE.  This is because the PMON binary must first have a default NVRAM image embedded into it before the EST header can be added.  Please see the PMON Boot Loader chapter for information on embedding NVRAM information into a PMON image.  Please see the Using visionCLICK/visionICE chapter for details on how to add EST headers so that the PMON image can be burned to Flash using a visionICE.

PMON Boot Loader

Introduction

PMON is an open-source MIPS boot loader ported for the BCM47xx. PMON has three main purposes:

 

  • PMON performs some hardware diagnostics upon power-up
  • PMON will act as a system bootloader upon power-up
  • PMON can be used as a means to upgrade OS and application images in the Flash

 

Additionally, PMON supports a serial command line console.  Using a terminal application on your PC such as HyperTerminal, one can connect to the BCM47XX-based platform to monitor the boot process, monitor the self diagnostics and perform upgrade functions.

PMON Environment Variables 

PMON has several variables which are maintained in the NVRAM portion of the Flash.  For more information on the NVRAM section and its use, please refer to the NVRAM section of this document. 

 

PMON provides commands to manipulate these environment variables as well as updating the NVRAM with new variable values. 

set

The ‘set’ command has three functions:

 

  • It can be used to view the entire list of environment variables
  • It can be used to add a new variable or modify an existing one
  • It can be used to write the current set of environment variables to the NVRAM

 

To view the list of current environment variables, issue the ‘set’ command without any parameters.

 

To add a new environment variable or to modify an existing one, use the format ‘set <varname> <value>’  For example, to change the ‘localip’ parameter to ’10.0.0.1’ issue the command ‘set localip 10.0.0.1’

 

To write the current set of environment variables to the NVRAM, issue the command ‘set nvram’

unset

The unset command will do exactly what it sounds like it will do.  It will remove an environment variable.  To make the changes to the environment variables permanent, issue the ‘set nvram’ command.

Creating a Default NVRAM Image

The NVRAM contains many variables that a customized for a particular board and should never be lost, even if there are problems with the Flash in the field.  These variables include the MAC address for all of the interfaces, the serial number of the platform, any manufacturing lot numbers, etc.  At boot time PMON performs a CRC check of this NVRAM configuration area and if this area is invalid or does not have any data, the default NVRAM image is loaded. To this end, a default NVRAM images is embedded into the PMON binary itself and is located at offset 0x400 in the image.  Currently the length of this area is limited to 8 Kbytes.

 

If no default NVRAM image is embedded into a PMON binary and the PMON binary is programmed into a BCM47XX-based platform, PMON will create a generic NVRAM image.  All generic NVRAM images will have the same MAC and IP addresses so boards programmed this way will conflict with each other.  These values can be changed from the PMON serial command line using the ‘set’ command described above.

nvserial

Along with the source code distribution for the BCM47XX, you received a utility (in the /tools directory) called NVSERIAL.  This utility is used to embed a default NVRAM image into the PMON boot loader binary file.

 

The majority of the default NVRAM information is stored in a plain text file.  Along with your distribution, you received two files:  R4_NVDATA.TXT and DEV_NVDATA.TXT.  These files are very similar but have some minor changes with respect to the SDRAM configuration.

 

The xx_NVDATA.TXT files are plain text ASCII files that have a list of NVRAM environment variables and default values.  The file should be edited so that the variables match the particulars of your BCM47XX-based platforms.  Most importantly, the ‘sdram_init’, ‘sdram_config’and ‘sdram_refresh’ variables must be configured to match the memory configuration of your particular platform. For more information, please refer to the BCM47XX Programmer’s Guide.In addition, the MAC address variables for the Ethernet and HPNA interfaces should be modified to be unique to the board being programmed.

 

Once you have modified the NVDATA.TXT information, you can use the NVSERIAL utility to embed a default NVRAM into the PMON binary.  To do this, enter the command:

 

nvserial -i <input file> -o <output file> -b 0x400 -c 0x2000 -s 42 nvdata.txt

 

This puts 8 KB of NVRAM data at a 1 KB offset with the serial number set to 42 in the default NVRAM data.

 

Of course, you can enter whatever you want for the PMON input and output file.  Also, you can specify a different filename instead of NVDATA.TXT.  In fact, with your distribution you received two files R4_NVDATA.TXT and DEV_NVDATA.TXT which correspond to values used for two of our reference platforms.

 

NVSERIAL can also be used to create an NVRAM image not embedded into PMON.  The purpose of this image would be to create an NVRAM image that would be burned directly into Flash using the visionICE Flash programming facilities.  To create this NVRAM image, enter the following command:

 

nvserial -o nvram_42.bin -s 42 r4_nvdata.txt

 

This would create an NVRAM image with the serial number 42 suitable for use on a BCM94710R4 platform. Please see the section titled Preparing a Flash Binary Image in the Using visionCLICK/visionICE chapter for information on adding the appropriate header and writing the NVRAM image to Flash.

Programming PMON Into Flash

Once you have a PMON binary image,   you can then embed a default NVRAM image as described above.  You are then ready to program the resulting image into Flash.

 

For programming the PMON image into the target Flash, one could use a standard Data I/O programmer or WindRiver VisionICE tool. Refer to the Programming a Flash Device section of the Using visionCLICK/visionICE chapter information on how to do the actual Flash programming. For using the Vision ICE tool, one needs access to the JTAG port of the BCM47xx in the target board. Once programmed, remove the visionICE pod from the target board, and recycle power for PMON to boot up.

Connecting a Serial Monitor to a BCM47XX-based Platform

You can connect via a serial port to the BCM47XX-based platform.  In all cases, you will need a straight-through serial cable to connect between the PC and the BCM47XX-based platform.  Depending on the platform, you may not need any other hardware to make the connection but for at least the BCM94710R4, you will need the Broadcom-supplied serial connector (assembly # 102371-00) to connect to the J9 jumper block.

 

Once you have made the physical connection, load a serial terminal application (such as HyperTerminal) on the PC.  Configure the connection to use these parameters:

 

  • 115K baud
  • 8 bits
  • 1 stop bit
  • no parity
  • no flow control

 

When you reset the BCM47XX-based platform, you will see output as the PMON boot loader executes.

When PMON Boots…

These are the steps that PMON goes through when it boots up:

 

  • First, it does self-diagnostics on the SDRAM, Ethernet and HPNA functions.  If there is an error, the appropriate error message is displayed on the serial console and the booting sequence continues.
  • Second, it waits for 3 seconds to see if a new OS image is to be loaded into Flash via the LAN Ethernet port
  • Third, it tries to boot a compressed OS image stored in the Flash location specified by the ‘os_flash_addr’ environment variable.
  • Lastly, if the OS boot fails, PMON returns to the state where it waits for a new OS image on the LAN Ethernet port

Erasing the NVRAM Area

It may be desirable to erase an existing NVRAM image in Flash and start over from scratch.  To do this, you can use the ‘nvErase’ command in PMON.  To do this, perform the following steps:

 

  • Connect a serial monitor to the BCM4710-based platform as described above
  • As PMOM boots, hit <CTRL-C> to enter the PMOM prompt
  • Enter the command ‘nvErase bffe0000 20000’ to erase the NVRAM in Flash
  • Reboot the system

 

When the system starts, PMON will see that the NVRAM is blank and write a new default NVRAM image into Flash.

Loading an OS Image Using PMON

Upon power on/reset PMON will wait for 3 seconds, listening on the LAN Ethernet port to see if a new OS image is to be transferred.  If no image is received, or an image is received with a corrupt/invalid header, PMON will then attempt to boot the OS already located in Flash.

 

NOTE:  An environment variable named ‘boot_wait’ exists which controls whether PMON should wait for an OS image upgrade at power-up or not.  If a developer wants quick boot times and uses other means (such as visionICE) to program the Flash, this variable can be set to ‘on.’  But if set to on, OS image updates at power-up will no longer be possible.

Adding a Header

A header for the binary images needs to be added using the TRX utility prior to sending it to PMON for download onto the target Flash.   PMON checks the header id, image length and crc for every downloaded image. If header is not found, the downloaded OS image will not be saved into the flash and not be auto-booted.  NOTE:  A TRX header does not need to be added to the PMON binary itself.

 

To add a header to a compressed OS image:

 

 

trx –o os_trx.bin os.bin

 

where

os.bin = original untagged imaged (input)

os_trx.bin = tagged/added image (output)

 

TRX can also create a single file out of multiple images.  For instance, both the Linux kernel and the Linux root file system can be concatenated into a single image that can be uploaded to the platform using PMON.  The following is an example:

 

trx -o linux.trx vmlinuz -b 768k rootfs

 

The resulting image will have the TRX header at offset 0 immediately followed by the Linux kernel.  At at offset of 768Kbytes into the image, the root filesystem will be located.  When any image is uploaded via PMON, it will begin at the 256Kbyte offset in Flash.  So in this example, the Linux file system will ultimately reside at the 1Mbyte (768 + 256) boundary in Flash.

To actually transfer the file to the BCM47XX-based platform, you will use the TFTP application.  TFTP comes standard on both Windows and Linux installations and needs to be invoked from a command-line prompt.  The format for the command in Windows is:

 

tftp –I <IP address> PUT <OS image file>

 

In Linux, do the following:

 

      prompt> tftp <IP address>

      tftp> binary

      tftp> put

      (file) <OS image file>   

 

The IP address of the router is defined by the ‘localip’ environment variable.  Once you enter the TFTP command, the image will be  downloaded to the target RAM starting at the location specified by the ‘dl_ram_addr’ PMON environment variable.  PMON will then immediately save this image into the Flash at the location specified by the ‘os_flash_addr’ variable.  It will then copy or decompress the image from Flash back into RAM and boot it.  Currently PMON assumes the maximum length of the OS image is 3 MB. 

Using visionCLICK/visionICE

Introduction

The combination of visionCLICK software along with a visionICE hardware debugger provides for a powerful debugging platform to be used with BCM4710-based designs. 

 

The visionCLICK/visionICE tools are available from WindRiver systems.  However, some additional files are required in order to fully utilize these tools with your BCM4710-based hardware design.  These additional files are provided along with the source code that is outlined in your Source Code License Agreement (SCLA).

 

The instructions in this document will guide you in preparing your visionICE environment to be used with your BCM4710-based platform as well as go over some additional uses above and beyond normal debugging.  Before continuing, please refer to the instructions from WindRiver on how to install the visionCLICK software on your Windows PC.

 

Preparing Your visionICE for Use with the BCM4710

After you have installed the visionCLICK software on your computer, you should ensure that you can communicate with the visionICE unit.  Communication between the visionCLICK software and the visionICE unit can be done through traditional computer ports (COM, LPT) or through an Ethernet connection.  We suggest that you use the Ethernet connection as it provides a couple of key benefits:  faster communication with the visionICE and ability for remote debugging of hardware platforms.

Configuring the IP Address Parameters

In order for visionCLICK to use the network (Ethernet) connection to communicate with the visionICE, the visionICE must first be assigned an IP address.  At this time, assigning an IP address using DHCP is not supported!  Please follow these steps to program a static IP address into the visionICE.

 

Connect a serial cable from your visionICE to the serial port on your computer (usually the COM1 port.)  An appropriate serial cable should have been provided with your visionICE hardware.  Once the physical connection has been made, open your terminal software (i.e. HyperTerminal for Windows) and use the following settings:

 

  • 8 data bits
  • 1 stop bit
  • no parity
  • no flow control
  • 9600 baud

 

When you connect with your terminal software, you will see a “NET>” prompt.  To configure the IP parameters of the visionICE, follow these steps:

 

  • Enter the command ‘ethsetup’. 
  • From the resulting menu, choose “2. Modify Basic IP Parameters.” 
  • Choose “1. Enter the IP Manually.”  Enter the IP address to be used for the visionICE
  • Choose “1. Enter the Netmask Manually.”  Enter the netmask to be used in hexadecimal format.  For instance, if your netmask is 255.255.255.0, enter 0xffffff00.
  • If you need to access the visionICE through a router, choose “Y” from the “Should this system use a default gateway” prompt.  Then enter the default gateway.
  • Next choose “8. Save parameters” and then “9. Exit setup mode”
  • Enter the ‘reset’ command to reset the ICE so that it uses these IP parameters

 

You should now be able to ping the visionICE at the IP address you setup.  If you cannot ping the visionICE, go through the above steps and verify the values you entered.

Debugging via a Terminal Application

We recommend using the visionCLICK software for debugging the platform.  If, however, you would like to enter some basic visionICE commands via a terminal application, you can do this.  From the ‘NET>’ prompt, enter the ‘bkm’ command.  This will allow you to enter commands like ‘dc’ to check the version, ‘cf’ to configure the project, etc.

 

In order to enter the BKM mode, the visionICE must be connected to the EJTAG port on the BCM94710 platform.  Be sure the pin ‘1’ on the the EJTAG connection is connected to pin ‘1’ on the BCM94710 platform.

 

When you enter the ‘bkm’ command, you will be presented with a different prompt.  This prompt can be either ‘BKM>’ or ‘ERR>’  Either of these prompts indicates that you are no longer in the ‘NET>’ configuration mode.  The difference is that the ‘BKM>’ prompt is presented when the visionICE has made a connection with the target hardware (i.e. the BCM94710R4)  The ‘ERR>’ prompt is presented when there is no connection.

 

At the BKM>/ERR> prompt, you can enter the ‘he’ command to get a list of options.  For more information, please refer to the documentation that accompanied your visionICE.

Verifying the Connection

Once all of the IP parameters have been properly set through the serial port, we are now ready to test the network connection between visionCLICK and the visionICE hardware.  To do this, perform the following steps:

 

  • Open the visionCLICK software.  When you installed visionCLICK, it should have put a shortcut on your Windows start menu.
  • On the dialog box that appears, choose the “Configure” option.
  • Click on the “Communications” tab. 
  • In the “Normal Port/Rate” section, chamge from “LPT1” to “Net”
  • Enter the IP address that you assigned the visionICE and click the “Connect” button.
  • You will see a “Warning: Configuration Conflict” dialog box.  Choose the “No” option.  We will create a new project file in a later section.
  • If all goes well, the status line should show “Connected...” in the status section near the top of the dialog box.  You will also be able to open the “Terminal” window in visionCLICK to issue commands directly to the visionICE.
  • Click “Ok” to close the configuration window.  Click “No” on the next dialog box as we will be creating a new project later.

 

Note:  By default, the visionCLICK software will try to initialize the target hardware to be debugged.  At this stage, it is not necessary that you have your hardware connected to the visionICE.  If you see a dialog box that complains that it is unable to make a connection to the target hardware, ignore this.  Check the “Do not display this error window” checkbox to avoid seeing this in the future.

Upgrading the visionICE Firmware

Main Firmware Upgrade

The firmware that was on your visionICE when you received it may or may not be compatible with the BCM4710.  To check the firmware level, you can use the ‘dc’ (display configuration) command in the Terminal window within visionCLICK.  You will see output similar to:

 

      >ERR>dc

      Firm Rev  = vn 1.8i

Big/Little Endian Matching

One of the configuration parameters in the BCM4710_R4.REG file is the endianess of the platform.  The LENDIAN setting in the BCM4710_R4.REG file must match the endian mode being used on your hardware platform.  Please perform the following:

 

  • Open up the BCM4710_R4.REG file with a text editor.  This will be located in \ESTII\REGFILES|MIPS.
  • Locate the line that reads “CF LENDIAN YES”
  • If your platform is operating in little endian mode, then no change is necessary.
  • If, however, your platform is operating in big endian mode, then you need to modify this line.  Change the line so that it reads “CF LENDIAN NO” and save the file.

 

For the BCM94710R4 board, there is a jumper, J10, that controls the endian mode of the processor.  When J10 is jumpered, the BCM4710 will operate in big-endian mode.  When J10 is not jumpered, it will operate in little-endian mode.

Programming a Flash Device

The visionCLICK software can be used to program the Flash device on your BCM4710-based design.  This is especially useful if you have a new board back with a completely blank Flash part.  In this instance, you can use the visionCLICK/visionICE to quickly get a Flash image up and running.

Getting Ready

Before you can program a Flash device, you must be able to initialize the visionICE and target platform.  To do this, issue the ‘in’ command in the terminal window.  If all goes well, the prompt will change to “BKM>” which indicates that you are now in background debug mode.

Using the SWAP Utility

If you need to program a big-endian Flash image using the visionICE while in little-endian mode, the images must first be run through the SWAP utility.  This utility is provided on your distribution in the /tools/bin directory.  The format of the command is ‘swap <filename>’  The output will have a ‘_bin’ embedded in the output file.  For example, to swap the ‘pmon.bin’ file:

 

       swap pmon.bin

 

The output will be a file called ‘pmon_swap.bin’  Use this swapped file in the following section on Preparing a Flash Binary Image.

Preparing a Flash Binary Image

First of all, if the image that you are to be writing to the Flash is an OS image that will be booted by the PMON boot loader then you will need to add a header that tells PMON the size of the OS image and contains a checksum to validate the image.  Without this header, PMON will not boot the OS image.  Please refer to the “PMON Boot Loader” section for information on how to use the TRX utility to do this

 

Next, in order for a Flash image to be programmed using the visionICE, it must  contain another header defined by WindRiver that instructs the visionCLICK software the size of the image and where in Flash it should be programmed. Broadcom has developed a program called ADDHDR (for both Win32 and Linux environments) that will add this header to a flat binary file.  (Please refer to the documentation on your specific build environment to obtain details on how to create a flat binary image.) NOTE:  The ADDHDR program should be run after any other utilites, such as TRX or NVSERIAL, have been run.  To use ADDHDR, simply issue the command

 

ADDHDR <filename>

 

where filename is the flat binary file you wish to prepend the header to.  The resulting output file will have a _EST suffix before any filename extension.  This _EST file will be used to program the Flash from within visionCLICK.

Basic Debugging

Debugging a Linux Image

Before you can debug a Linux image, you must disable the watchdog timer.  In the Linux platform, the watchdog timer is setup so that if there is no processor activity for a specified period of time, the platform is reset.  The intent here is that if some unknown bug exists which causes the router to lock up, it will automatically reset and allow the user to continue to use the device, sometimes without ever noticing anything went wrong.

 

This is desirable in the field, but causes problems with debugging.  When the visionICE tries to take control of the processor, the wathdog timer thinks that the platform has crashed and causes it to reset, thus terminating the link between the visionICE and the platform.  To disable the watchdog timer to allow debugging via visionICE, perform the following steps:

 

  • Connect a PC running a terminal application to the serial connector on the BCM94710 platform.
  • The serial parameters are:  115000 baud, 8 bits, 1 stop bit, no parity, no flow control
  • At the terminal prompt, enter the command “nvram unsetenv watchdog”
  • Reset the 94710.

 

The watchdog timer is now disabled and the visionICE can be used to debug the linux kernel.

The NVRAM

Introduction

In VxWorks and Linux router designs based around the BCM4710, the last 32Kbytes of Flash are designated as non-volatile RAM (NVRAM) space.  This NVRAM is used as a storage space for variables that are used by the PMON boot loader as well as the router image. 

NVRAM Protection

The NVRAM section of the Flash should be preserved, even if new router images are being programmed into the board.  This is because the NVRAM will contain vital information such as the MAC addresses for each of the interfaces as well as the serial number and any other vendor specific identifying information, such as lot number or manufacture date.  This information, if lost, may be difficult or even impossible to reconstruct.  Even if this information is easily retrievable (perhaps from the boot loader itself) other information such as the current router IP configuration and any statically forwarded ports should be preserved if at all possible.  Loss of this user-configured information would lead to an undesirable user experience.

Accessing NVRAM Information

Don’t Write NVRAM Data Directly

In order to maintain NVRAM integrity and to be sure that all clients of the NVRAM data (PMON, VxWorks, Linux) can properly access data stored in the NVRAM, the NVRAM memory block must NOT be modified directly.  Direct modification of the data in the NVRAM section will result in an unusable NVRAM. 

How the NVRAM Is Accessed

To ensure that all clients of the NVRAM information are in sync, each client is compiled with shared NVRAM access libraries.  These libraries deal with reading of NVRAM environment variables, changing or creating new variables, calculating checksums, writing NVRAM to Flash, etc. 

Getting an Initial NVRAM Environment

A default NVRAM image can be inserted into a PMON boot loader file.  For instructions on how this is done, please refer to the PMON Boot Loader section.

 

If a PMON image with default NVRAM is booted on a board, it will check to see if the NVRAM is either missing entirely or is corrupted.  If either is true, the default NVRAM is loaded into the NVRAM section of the Flash and is subsequently used by PMON and any OS image loaded into the Flash.

Reading/Writing NVRAM Environment Variables

In general, NVRAM environment variables are read/written to by any of the NVRAM clients.  For instance, if the Linux router is running and a configuration change was made that altered the IP address to be used on the WAN port, the Linux code would write this new information to the NVRAM memory.  Also, if some information like a static route entry was no longer needed, the Linux code would remove this variable from the NVRAM space.  (All of this NVRAM manipulation is done using the NVRAM access libraries described above.)

 

If you would, however, like to view the NVRAM variables or create new ones, this can be done with the PMON boot loader.  Simply connect a serial client to the BCM4710-based board and reset it.  You will see the PMON boot information displayed.  You should quickly hit <CTRL-C> to break the PMON boot process and get to the PMON prompt.  From here you can enter the ‘set’ command to see all of the current environment variables.  For more information on PMON NVRAM manipulation commands, please refer to the PMON Environment Variables section of the PMON Boot Loader chapter.

NVRAM Format

NVRAM Header

The first 40 bytes of the Flash spaced designated as the start of NVRAM are is the NVRAM header.  The format of the NVRAM header is as follows:

 

struct nvram_header {

        unsigned long magic;

        unsigned long len;

        unsigned long crc_ver_init;

        unsigned long config_refresh;

        unsigned long reserved;

};

 

The definition for each of the structure members is:

 

  • Magic:  Unique identifier marking the beginning of the NVRAM data sector.  The value of this identifier is 0x48534C46, or ‘FLSH’ in ASCII.
  • Len:  The total length of the NVRAM including the NVRAM header and all of the NVRAM environment strings.
  • Crc_ver_init:  Bits 0-7: NVRAM CRC.  Bits 8-15: NVRAM version.  Bits 16-27:  Value to be written into the SDRAMInit regieter.  Bits 29-31: reserved.
  • Config_refresh:  Bits 0-15: Value to be written into the SDRAMConfig register.  Bits 16-32: Value to be written into the SDRAMRefresh register.

 

NVRAM Environment Variables

The NVRAM environment variables are located directly after the NVRAM header.  The environment variables are represented as a series of NULL-terminated strings.  The strings are in the format

The end of the strings is deliminated by two consecutive NULL bytes.

Set to restore to factory defaults on the next reset.  ‘0’ if no restore required, ‘1’ to perform restore.
